# QueryForge — обзор сервиса

Документ описывает назначение, механизм работы и пользу сервиса. Цель — сделать очевидным, какие проблемы решает QueryForge и почему он полезен.

---

## 1. Модели в Cursor: сильные и слабые

QueryForge полезен при работе со **слабыми** моделями. В Cursor:

| Тип | Режимы и модели | Поведение |
|-----|-----------------|-----------|
| **Сильные** | **Chat:** Claude Sonnet, Claude Opus, GPT-4/5, Gemini Pro — выбираешь сам | Понимают размытые формулировки, сами разбивают сложные задачи, «додумывают» контекст. QueryForge не обязателен. |
| **Слабые** | **Composer**, **Agent** — проприетарные модели Cursor | Быстрее и дешевле, но нужны явные инструкции, разбиение на шаги, указание файлов. QueryForge превращает сырой запрос в подзапросы. |
| **Auto** | Cursor сам выбирает модель | Непредсказуемо: может взять Sonnet (сильная) или Composer (слабая). При слабой — QueryForge полезен; при сильной — можно писать свободно. |

---

## 2. Какую проблему решает сервис

### Ситуация

Ты привык формулировать запросы **свободно** — как для сильной модели: с неявным контекстом, без детализации, полагаясь на то, что модель «додумает». Сильные модели хорошо понимают размытые формулировки и сами разбивают сложные задачи.

**Слабые модели** (Composer, Agent) — быстрее и дешевле, но:
- лучше работают с **явными** инструкциями;
- хуже справляются с очень размытыми или многошаговыми запросами «одним куском»;
- могут «потерять нить» в длинных задачах без чёткого плана.

### Проблема

**Без QueryForge:** ты даёшь слабой модели сырой запрос. Модель пытается выполнить его целиком. Результат непредсказуем: иногда хорошо, иногда — пропущенные шаги, неверный порядок действий, потеря фокуса.

**С QueryForge:** ты даёшь тот же сырой запрос. Сервис превращает его в набор **готовых подзапросов** с явными инструкциями, контекстом и критериями готовности. Модель выполняет по одному — результат стабильнее и ближе к тому, что дала бы сильная модель.

## 3. Механизм работы

### Схема

```
┌─────────────────────────────────────────────────────────────────┐
│  Твой запрос (свободный, как для сильной модели)                 │
│  + опционально: контекст проекта (CONTEXT.md)                    │
└─────────────────────────────┬───────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  QueryForge (OpenAI GPT-4o)                                      │
│  • Классифицирует задачу (Feature/Refactor/Infra/Bugfix)         │
│  • Оценивает: нужна ли декомпозиция?                             │
│  • Разбивает на подзадачи с risk_level, rollback_hint, depends_on │
│  • Добавляет expected_artifacts, verify_hint                     │
│  • Возвращает risks_covered, missing_risks (Auto-plan diff)       │
└─────────────────────────────┬───────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  Выход: composer_queries[]                                        │
│  • query, context, done_when                                      │
│  • risk_level, rollback_hint, depends_on (Execution Safety)        │
│  • expected_artifacts, verify_hint (наблюдаемый план)              │
│  • analysis: task_type, risks_covered, missing_risks (Auto-plan)    │
└─────────────────────────────┬───────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  Ты копируешь каждый подзапрос в Composer/Agent (в проекте)      │
│  Модель выполняет по одному                                      │
└─────────────────────────────────────────────────────────────────┘
```

### Ключевые решения QueryForge

1. **Не дробить лишнее** — если задача простая (модель найдёт нужное через semantic search), возвращается один запрос.
2. **Не перегружать @** — Composer/Agent сам ищет файлы. @ указываются только для ключевых путей из контекста проекта.
3. **Разбивать только сложное** — архитектурно сложные задачи, неясные требования, много несвязанных модулей.
4. **Execution Safety** — каждый шаг имеет risk_level, rollback_hint, depends_on.
5. **Наблюдаемый план** — expected_artifacts и verify_hint для проверки «появилось ли».
6. **Auto-plan diff** — risks_covered и missing_risks визуализируют ценность декомпозиции.

## 4. Сравнение: с QueryForge и без

Один и тот же запрос: *«Добавь OAuth через Google, чтобы пользователи могли входить через аккаунт»*

### Вариант A: Слабая модель без QueryForge (сырой запрос)

| Этап | Что происходит |
|------|----------------|
| **Ввод** | Ты вставляешь запрос как есть в Composer/Agent |
| **Поведение модели** | Пытается выполнить всё за один проход. Ищет auth-файлы, добавляет OAuth. |
| **Риски** | Может пропустить настройку env, миграции, обновление UI. Может выбрать не тот паттерн (если в проекте уже есть другой OAuth). Может «забыть» про тесты или документацию. |
| **Результат** | Непредсказуем. Часто нужны доработки и уточняющие запросы. |

### Вариант B: Слабая модель с QueryForge (разбитые запросы)

| Этап | Что происходит |
|------|----------------|
| **Ввод** | Ты даёшь тот же запрос в QueryForge (с контекстом проекта) |
| **Выход QueryForge** | Например: (1) Изучи структуру auth в @backend/auth/, добавь OAuth-провайдер Google. (2) Добавь env-переменные для client_id/secret. (3) Обнови UI входа — кнопка «Войти через Google». (4) Добавь тесты для OAuth flow. |
| **Поведение модели** | Выполняет по одному запросу. Каждый — с явной целью и критерием готовности. |
| **Преимущества** | Чёткий порядок. Ничего не забыто. Можно проверять после каждого шага. Легче откатить один шаг при ошибке. |
| **Результат** | Стабильнее, ближе к тому, что сделала бы сильная модель. Меньше итераций и правок. |

### Сводная таблица

| Критерий | Без QueryForge | С QueryForge |
|----------|----------------|--------------|
| **Формулировка** | Сырой запрос | Явные подзадачи с критериями |
| **Порядок действий** | Модель решает сама, может ошибиться | Задан явно |
| **Контекст (@файлы)** | Модель ищет сама | Ключевые пути указаны (из CONTEXT.md) |
| **Критерий готовности** | Неявный | Явный (done_when) |
| **Риск пропустить шаг** | Выше | Ниже |
| **Итерации после выполнения** | Чаще | Реже |
| **Предсказуемость** | Ниже | Выше |

**Наблюдаемые эффекты** (что можно замерить): меньше rollback, меньше повторных запросов, меньше hallucinated решений (когда модель «придумывает» несуществующие файлы или паттерны).

## 5. Когда QueryForge даёт наибольшую пользу

- **Сложные фичи** — OAuth, платежи, миграции, рефакторинг нескольких модулей.
- **Неочевидный порядок** — когда шаги зависят друг от друга и порядок важен.
- **Проект-специфика** — когда нужны точные пути и стек из CONTEXT.md.

### Когда QueryForge почти не нужен

- Простые задачи («добавь кнопку», «исправь опечатку»).
- Задачи с чётким scope, где модель сама найдёт нужные файлы.

В таких случаях QueryForge вернёт один уточнённый запрос (`needs_decomposition: false`) — без лишнего дробления.

---

## 6. Итог: почему сервис полезен

1. **Один стиль формулировок** — пишешь как для сильной модели, получаешь запросы, подходящие для Composer/Agent.
2. **Меньше итераций** — явные подзадачи и критерии снижают число доработок.
3. **Контроль** — видишь план до выполнения, можешь скорректировать или пропустить шаг.
4. **Контекст проекта** — CONTEXT.md даёт реальные пути и стек, а не абстрактные `src/auth/`.
5. **Без лишнего дробления** — простые задачи не разбиваются, сложные — разбиваются осмысленно.
6. **Обучающий эффект** — по мере использования новички привыкают к структуре запросов, языку и экономии токенов.

---

## 6.1. QueryForge как обучающий сервис

**Идея:** QueryForge улучшает запросы — и как побочное действие **обучает**, как должны выглядеть хорошие запросы для Cursor.

| Что видит пользователь | Чему учится |
|------------------------|-------------|
| Сырой запрос → разбитый план | Структура: один шаг = одна цель, явный done_when |
| @ только ключевые пути | Не перегружать контекст, Composer/Agent найдёт остальное сам |
| risks_covered / missing_risks | Какие риски важно явно учитывать (env, UI, тесты) |
| expected_artifacts, verify_hint | Проверять «появилось ли», а не «сделал ли» |
| Простые задачи не дробятся | Когда модель справится сама — не усложнять |

**Результат:** со временем пользователь начинает писать запросы в «правильном» формате сам, экономит токены и не перегружает Cursor лишним контекстом.

## 7. Надёжность сервиса

| Параметр | Значение |
|----------|----------|
| **Retry** | 3 попытки при APIConnectionError, APITimeoutError, RateLimitError (экспоненциальная задержка 2s, 4s, 8s) |
| **Timeout** | 120 с на вызов OpenAI API |
| **Лимиты ввода** | Запрос до 50 000 символов, контекст до 100 000 |
| **Health check** | `GET /health` → `{"status": "ok"}` (Docker healthcheck каждые 30 с) |
| **Обработка ошибок** | Валидация ответа API, graceful fallback при сбоях |

## 8. Реализованные улучшения (решение слабых мест)

| Проблема | Решение | Статус |
|----------|---------|--------|
| **Ценность скрыта** | Auto-plan diff в UI: «Без декомпозиции» vs «С декомпозицией», risks_covered / missing_risks | ✅ |
| **Ценность скрыта** | Демо-режим: кнопка «Демо (подписка)» — пример без API | ✅ |
| **done_when декларативный** | `expected_artifacts` + `verify_hint` — проверять «появилось ли», а не «сделал ли» | ✅ |
| **Execution Safety** | `risk_level`, `rollback_hint`, `depends_on` на каждый шаг | ✅ |
| **Типы задач** | `task_type`: Feature / Refactor / Infra / Bugfix | ✅ |

### Остаточные риски

1. **Зависимость от модели** — QueryForge остаётся **Prompt Transformer**. **Execution Planner** (управление выполнением) требует API Composer/Agent — у Cursor его пока нет.

2. **Валидация вручную** — `expected_artifacts` и `verify_hint` дают подсказки, но проверку выполняет человек. Автоматическая валидация (скрипт проверки) — возможное развитие.
